# AGENTS.md

## Scope

本文件适用于仓库根目录下的全部文件。它只记录长期有效的协作流程、工程规矩和代理工作方式；临时进度、详细设计、任务拆解和方案背景不要堆在这里，优先放到 `docs/`、README 或专门的任务文档中。

## Working Flow

- 开始前先快速了解当前任务、相关文件和已有实现，避免凭空改动。
- 改动前说明接下来要做什么；较复杂任务先列出简短计划，再逐步更新进度。
- 优先做最小、直接、可验证的修改，尽量从根因解决问题，避免顺手重构无关代码。
- 修改功能、修复问题或调整布局后，同步检查是否需要更新 `README.md`、`docs/design.md`、`docs/pitfalls.md`、源码注释和相关 skill 草案；不要只改代码不更新上下文。
- 修改后尽量运行与改动最相关的构建、测试或静态检查；如果无法验证，要在交付说明里写清原因。
- 最终回复只总结关键改动、验证结果和必要的后续建议，不要粘贴大段已写入仓库的代码。

## Repository Rules

- 保持现有 Qt Widgets / CMake 项目结构，不随意移动文件或改名。
- 新增工程设计、功能说明、阶段进度、截图说明等内容时，优先写入 `docs/`；`AGENTS.md` 只保留稳定规则。
- 遵循现有 C++/Qt 风格：命名、布局、信号槽连接方式和 UI 文案处理保持一致。
- 代码注释遵循“中文优先、清晰必要、随代码同步维护”的原则；具体要求见 Code Comments。
- 打包、构建、检查脚本优先放在 `scripts/`；发布产物、安装包、便携包和本地构建输出不要纳入版本库。
- 不要主动提交 git commit、创建分支或改动无关文件，除非用户明确要求。

## Git Commit Messages

- 用户要求生成 Git 提交日志时，默认输出可直接粘贴到 TortoiseGit 的纯文本，不使用 Markdown 代码块或表格。
- 提交日志内容要写入 `build/commit-message-<版本号>.txt`，再在回复中给出文件路径；`build/` 已被忽略，不会进入版本库。
- 每次提交前都要检查应用版本号，当前项目版本号集中在 `include/appversion.h` 的 `AppVersion` 常量中；首次仓库基线提交不递增版本，保持既定初始版本号。
- 提交日志需要使用中英双语，包含清晰摘要、关键变更、验证情况和版本号/版本说明；中英文内容要分成相互独立的两段或两个小节，不要一句英文一句中文交错混排。

## Code Comments

- 新增或修改 C++/Qt 源码时，注释优先使用中文，并保持与现有文件风格一致。
- 新增源码/头文件要写文件头注释，包含文件名、版本、描述、作者、日期和必要变更说明。
- 文件头 `Change Log` 同一天只保留一个版本条目；同一天内的多个小修复要合并描述，详细历史放到 README、`docs/design.md` 或 `docs/pitfalls.md`，不要在文件头连续堆 V1.x。
- 新增非平凡函数要写函数注释，说明用途、关键参数和返回值；修改函数行为时同步更新已有函数注释。
- 行内注释用于解释协议细节、边界条件、平台限制、UI 布局原因等不直观逻辑，不为显而易见语句重复注释。
- 涉及中文注释或文案时，读取和写入文件必须显式使用 UTF-8，避免后续代理看到乱码。

## Shell And Encoding

- Windows 下优先使用 PowerShell 7：`pwsh.exe -NoProfile -Command`。
- 避免使用 Windows PowerShell 5.1：`powershell.exe`，除非 `pwsh.exe` 不可用或任务明确需要。
- 读取和写入文本/源码时显式使用 UTF-8，例如 `Get-Content -Encoding UTF8`，避免中文乱码。
- 搜索文件和文本优先使用 `rg`；如果不可用，再使用 PowerShell 原生命令。

## Sub-Agent Usage

- 默认使用子代理提升效率；只有用户明确要求不要使用子代理、不要并行或只由主代理处理时，才停用子代理。
- 子代理只承担独立且边界清晰的任务，优先用于代码库探索、独立模块实现、并行验证等不阻塞当前主线的工作。
- 分配子代理时要说明任务目标、可修改范围、预期输出，以及不要回滚他人改动。
- 不要把当前必须马上完成的阻塞性工作交给子代理后空等；主代理应继续处理不重叠的主线工作。

## Pitfall Learning

- 开发过程中遇到构建、编码、平台、Qt/CMake、串口通信或工具链相关坑点时，要在交付总结中简要记录现象、原因和规避方式。
- 如果某类坑点会重复出现，先沉淀到 `docs/` 下的经验或排障文档，保持上下文可追溯。
- 当经验已经稳定、可复用、能指导后续代理开发时，再总结成 Codex skill；仓库内草案优先放在 `docs/skills/<skill-name>/SKILL.md`，skill 应保持精简，只放触发条件、核心流程、关键检查和必要脚本/引用。
- 不要把临时踩坑记录直接塞进 `AGENTS.md`；这里仅保留“需要复盘并最终沉淀为 skill”的规则。

## Documentation Guidance

- `README.md`：放面向用户的功能概览、构建方式和使用入口。
- `docs/design.md`：放架构设计、模块边界、重要设计决策和长期规划。
- `docs/` 下的新文档：放阶段性方案、调研记录、工作进度、截图说明或较长任务拆解。
- `AGENTS.md`：只放代理执行规则、协作约定和稳定工程约束。
