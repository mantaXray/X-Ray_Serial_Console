# 踩坑记录

本文件记录开发过程中已经遇到、可能重复出现、值得后续代理参考的坑点。稳定且高频的经验可以再沉淀为 Codex skill；临时任务进度不要写在这里。

## 2026-04-24：Windows PowerShell 5.1 读取 UTF-8 中文乱码

- 现象：用 `powershell.exe` 读取 `.cpp` / `.h` / `.md` 文件时，中文注释和文案显示成乱码。
- 原因：Windows PowerShell 5.1 默认编码/控制台代码页容易和项目 UTF-8 文件不匹配；PowerShell 7 默认 UTF-8，表现更稳定。
- 规避：Windows 下优先使用 `pwsh.exe -NoProfile -Command`，读取文本时显式加 `-Encoding UTF8`。
- 例子：`Get-Content -LiteralPath 'README.md' -Encoding UTF8`。
- 后续沉淀：如果后续仍频繁遇到 Windows 编码、中文文案或 Qt 翻译文件乱码问题，可整理成专门的 Windows/Qt 编码处理 skill。

## 2026-04-24：AGENTS.md 不适合承载详细进度

- 现象：容易把工作流程、设计方案、踩坑记录、开发进度都塞进 `AGENTS.md`。
- 原因：`AGENTS.md` 会被后续代理长期读取，内容过多会污染上下文，也会让稳定规则和临时信息混在一起。
- 规避：`AGENTS.md` 只写长期规则；设计放 `docs/design.md`，踩坑放 `docs/pitfalls.md`，阶段计划/调研另建 `docs/` 文档。
- 后续沉淀：如果项目文档类型继续增加，可整理一份项目文档分层 skill 或模板。

## 2026-04-24：沙箱下 Qt `moc.exe` 可能无法启动

- 现象：使用 `D:\QT\Tools\CMake_64\bin\cmake.exe --build build\Desktop_Qt_6_11_0_MinGW_64_bit-Debug` 时，CMake 进入 AutoMoc 阶段后报 `libuv process spawn failed: operation not permitted`。
- 原因：当前 Codex 沙箱限制了外部 Qt 工具链子进程启动，尤其是 `D:\QT\6.11.0\mingw_64\bin\moc.exe`。
- 规避：在 Codex 内运行 `pwsh.exe -NoProfile -File scripts\syntax-check.ps1` 做 C++ 语法检查；需要完整构建时，在用户本机终端或 Qt Creator 中运行 `pwsh.exe -NoProfile -File scripts\build-local.ps1`。
- 后续沉淀：如果后续频繁验证 Qt 项目，可整理 Qt/CMake 沙箱验证 skill，区分“代码编译错误”和“沙箱阻止工具链启动”。

## 2026-04-24：右侧 Dock 挤压主界面会导致控件重叠

- 现象：打开 Modbus 工具后，如果作为右侧 dock 停靠在主窗口内，会压缩接收区和底部控制区，串口参数、发送面板等控件出现重叠。
- 原因：主窗口底部控制区信息密度高，右侧 dock 占用水平空间后，部分固定宽度/紧凑布局没有足够空间重排。
- 规避：大型工具窗口默认使用浮动 `QDockWidget`，并设置 `setAllowedAreas(Qt::NoDockWidgetArea)`、合理的 `resize` / `minimumSize`，避免重新停靠挤压主界面。
- 后续沉淀：如果继续增加大型工具面板，应优先设计为浮动窗口或独立对话框，只有轻量面板才考虑停靠到主界面。

## 2026-04-24：Modbus 表格行数增加后出现大片空白

- 现象：Modbus 读写工具把读取数量从默认值调大后，表格区域下方出现明显白色空白，看起来像数据行没有刷新出来。
- 原因：表格和日志共用纵向 splitter，日志区会抢占高度；同时刷新行数后反复 `resizeColumnsToContents()` 会触发布局重算，容易出现空白闪烁或可视行不足。
- 规避：寄存器表格使用固定行高、设置最小可视高度、固定列宽策略，并让日志区域默认占用更小高度；刷新数据时不要反复全表自动撑列。
- 后续沉淀：以后做表格型实时监视窗口时，优先先确定“可视行数、滚动策略、splitter 默认比例、列宽策略”，再接入实时刷新逻辑。

## 2026-04-24：PowerShell 字符串中变量后接冒号会解析失败

- 现象：在双引号字符串里写 `"$path:$start-$end"` 会报 `Variable reference is not valid`。
- 原因：PowerShell 会把 `$path:` 解析成带作用域/命名空间前缀的变量引用，冒号后不是合法变量名时就会失败。
- 规避：变量后面紧跟冒号、字母或路径字符时，用 `${path}:$start-$end` 明确变量边界。
- 后续沉淀：如果后续 Windows 脚本坑继续增加，可整理成 PowerShell 编码、路径和字符串插值 skill。

## 2026-04-24：不要把本地缓存和被忽略截图写进 CMake 源列表

- 现象：提交前发现 `.vs/` 本地缓存处于未跟踪状态，同时 `CMakeLists.txt` 引用了 `docs/reference-images/` 下的本地截图。
- 原因：IDE 缓存和开发参考截图只适合本地使用；如果被忽略的文件写进 CMake 源列表，干净检出后可能因为文件缺失导致配置或构建失败。
- 规避：`.vs/`、截图和参考图统一进 `.gitignore`；CMake 源列表只放必须参与构建或需要随仓库分发的文件。
- 后续沉淀：提交前查看 `git status --short --ignored`，确认本地缓存、截图、构建产物没有混入待提交内容。

## 2026-04-28：隐藏的 Modbus 扫描与接收会话会影响主界面

- 现象：测试 Modbus 读写工具后关闭窗口，再使用主面板发送或轮询发送，主接收区看起来没有任何显示；进一步观察会发现设备回包还在进入 Modbus 的接收缓冲或原始日志。
- 原因：Modbus 读写工具是浮动 `QDockWidget`，关闭实际是隐藏；如果扫描定时器、等待响应状态或接收缓冲没有在隐藏边界清理，工具窗口会继续参与串口收发链路。仅用窗口可见性判断也不够，Modbus 日志会变成全局串口监听器。
- 规避：工具窗口 `visibilityChanged(false)` 时必须停止独立扫描定时器、取消等待响应状态、停止响应超时定时器、清理监视缓存、复位扫描按钮状态，并刷新工具状态。Modbus 接收处理必须由“工具刚发出请求且正在等待响应”的状态门控，响应完成、超时、停止扫描、改参数和清空数据都要退出该状态。
- 后续沉淀：凡是“隐藏不销毁”的 Qt 工具窗口，只要内部有定时器、后台发送任务或协议接收缓冲，都要在隐藏边界显式停止；协议工具的原始日志不要默认监听整条串口。

## 2026-04-28：超时分包与轮询发送组合会让接收区长期不刷新

- 现象：接收区开启“超时分包显示”后，主面板轮询发送期间，接收计数可能增加但接收文本迟迟不显示。
- 原因：超时分包依赖“接收静默超过 N ms”再刷新；轮询发送和设备连续响应会不断重启超时计时器，缓存一直不落到接收区。
- 规避：所有发送入口记录发送日志前，先刷新上一包待显示接收缓存，把发送动作当成自然的帧边界。
- 后续沉淀：串口 UI 中只要引入“按超时聚合”的缓存，就要同时定义发送边界、暂停刷新、关闭窗口等缓存刷新/丢弃策略。

## 2026-04-28：筛选关键字会让接收区看起来空白

- 现象：底部 S/R 计数增加，发送和接收内部都有数据，但接收显示区没有任何可见文本。
- 原因：`receiveFilterKeyword` 会在 `refreshReceiveView()` 中过滤显示行；如果关键字没有命中当前 TX/RX 文本，最终显示文本就是空字符串。配置文件里曾保存 `filter_keyword=USART1`，导致普通发送回显全被过滤。
- 规避：筛选生效时在接收区标题显示“筛选中”，过滤后无匹配内容时用空白提示说明原因；排查“计数增加但无显示”时优先检查暂停刷新、筛选关键字和超时分包。
- 后续沉淀：凡是显示层过滤不影响底层计数，都必须在 UI 上有明确状态反馈，否则用户会误判为串口接收链路断开。

## 2026-04-28：Modbus 日志选项不能直接改主接收区显示选项

- 现象：打开 Modbus 工具后调整原始日志的时间戳、超时分包或超时值，关闭工具回到主界面后，主接收区仍可能按这些选项缓存或显示，造成主页面发送后“无显示”的错觉。
- 原因：Modbus 原始日志控件曾直接连接到主接收区的 `timestampCheckBox`、`packetSplitCheckBox` 和 `receiveTimeoutSpinBox`，两个面板共用显示状态。
- 规避：Modbus 原始日志使用自己的控件和 `modbus/log_*` 配置项；`appendModbusMonitorLog()` 只读取 Modbus 日志选项，主接收区只读取 `display/*` 选项。Modbus 扫描发送请求时，工具 TX 日志不能复用会提前 flush 主接收区缓存的通用发送日志入口，否则主接收区开启“超时分包显示”时会被协议工具打断，无法按自己的超时窗口聚合 RX。
- 后续沉淀：两个视图看起来有相同选项时，也要先确认它们是不是同一显示对象；如果不是，不要为了省状态同步而直接共享控件或配置键。

## 2026-04-28：协议工具 TX 日志不能破坏主接收区超时分包

- 现象：主接收区开启“超时分包显示”后，Modbus 扫描每次发送请求都会让主接收区提前显示上一段 RX，最终主接收区的分包边界跟着 Modbus TX 走，而不是跟着自身接收静默超时走。
- 原因：主发送区、多字符串和普通工具发送需要在发送日志前 flush 主接收区缓存，把本次发送当作自然帧边界；但 Modbus 扫描有自己的原始日志视图，若复用同一个 TX 日志入口，就会把协议工具内部日志策略泄漏到主接收区。
- 规避：协议工具如果有独立原始日志，应区分“主接收区交通日志”和“工具内部日志”。Modbus 扫描 TX 可以显示在主交通日志里，但必须禁用发送前刷新主接收缓存；Modbus RX 在主接收区仍由自身超时分包定时器落屏，工具原始日志用自己的缓存和定时器。
- 后续沉淀：设计串口协议工具时，先列出哪些发送动作属于主接收区可见边界，哪些只是工具内部会话日志；不要因为两个日志格式相似就共享会改变显示时序的函数。

## 2026-04-28：配置保存不能隐式刷新主接收区超时分包

- 现象：打开 Modbus 工具、调整 Modbus 原始日志选项、拖动 Modbus 分隔条或表格列宽后，主页面接收区正在等待超时聚合的数据会提前落屏，看起来“接收超时”不生效。
- 原因：`saveDisplaySettings()` 原本会无条件调用 `flushPendingReceivePacket()`；而 Modbus 窗口可见性、splitter、表头、日志选项和默认格式变化都会保存配置，导致持久化动作改变了主接收区分包边界。
- 规避：配置保存函数必须保持纯持久化，不刷新或清空显示缓存。真正需要结束分包的边界由调用方显式处理，例如关闭程序、导出接收文本、切换主接收区显示格式或主发送链路发送前。
- 后续沉淀：命名为 `save*Settings()` 的函数不要带 UI 时序副作用；如果必须刷新缓存，单独放在调用点并写明原因。

## 2026-04-28：Modbus 表格、日志和顶部控件要按真实内容定默认尺寸

- 现象：Modbus 读写工具首次打开时，原始收发日志区域偏矮，需要滚动才能看清扫描 TX/RX；寄存器表有无意义空白；功能码等顶部控件比内容宽很多，或控件收紧后右侧留下大片空白。
- 原因：splitter 默认比例、表格最小高度、行高和输入控件宽度没有按真实内容反推，而是沿用了通用表单或表格的宽松默认值。把字段控件改成固定宽度后，如果父级 `QHBoxLayout` 仍用尾部 `addStretch()`、Dock 默认宽度也不调整，空白只会从控件内部转移到行尾。
- 规避：原始日志是扫描排障的核心区域，默认应给更高占比；寄存器表要用协调的最小高度、固定行高和列宽策略保留必要可视行数；功能码下拉框、站号、地址、数量、扫描间隔等控件按典型字符数和最大值设置宽度，`QSpinBox` 要把步进按钮和最大位数一起算进去；标签和输入框应作为字段组靠在一起，默认窗口宽度、最小宽度和持久化几何版本要随字段组总宽同步调整。
- 后续沉淀：Qt Widgets 工具窗口的默认布局要同时检查 splitter 初始比例、表格行高/最小高度、列宽总和、控件最大宽度和用户持久化状态；布局默认值变化时注意版本化或兼容旧配置。

## 2026-04-28：Modbus 修复验证不必每次跑完整构建

- 现象：小范围 Modbus UI/时序修复如果每次都跑完整构建，容易被沙箱下 Qt 工具链子进程限制阻断，验证成本高且和改动风险不匹配。
- 原因：当前环境下完整 CMake build 可能卡在 AutoMoc 或 Qt 工具启动权限；而这类修复更需要确认 C++ 语法、CMake 配置、关键回归文本和空白/编码问题。
- 规避：按用户要求，本轮验证策略改为运行语法检查、CMake configure、静态回归检查和 `git diff --check`，不运行完整构建。需要最终发布或二进制验收时，再由用户在本机普通终端或 Qt Creator 中跑完整构建。
- 后续沉淀：验证策略要和改动风险、环境限制匹配；不能把“沙箱完整构建失败”误判成代码一定失败，也不能在用户明确禁止时主动跑完整构建。

## 2026-04-28：固定单位不要放进数字输入框内部

- 现象：接收超时、循环发送间隔、Modbus 扫描间隔等 `QSpinBox` 使用 `setSuffix(" ms")` 后，输入框内部被单位占用，看起来拥挤，而且容易让人误以为单位可以切换。
- 原因：这些单位是固定毫秒值，不属于用户可编辑数据；把单位放进编辑框会混淆“数值”和“说明”。
- 规避：固定单位使用输入框外侧 `QLabel` 展示，例如“超时 50 ms”；禁用输入框时同步禁用单位标签。
- 后续沉淀：做 Qt 表单时先区分“值、单位、说明、模式选择”。只有单位本身可变时才做成输入控件的一部分或独立下拉框。

## 2026-04-28：表格默认列宽不能依赖用户发现横向滚动条

- 现象：多字符串面板首次打开时，表格列宽总和大于右侧面板默认宽度，部分列藏在横向滚动条后；同时复选框列只写 `HEX`，新增空白项默认按 HEX 发送，用户容易找不到切换位置。
- 原因：Qt `QTableWidget` 使用 `Interactive` 表头时不会自动压缩所有列；分隔条首次比例、dock 最小宽度和表头文案没有一起设计。无文字复选框列如果表头过短，也缺少足够的语义提示。
- 规避：默认列宽要按首次可见面板宽度倒推，必要时同步调整 splitter 默认尺寸和 dock 最小宽度；可持久化表头仍优先恢复用户拖拽状态，但默认列宽策略变更时要给表头状态加版本号，避免旧默认值永久覆盖新布局。复选框列要用明确表头、tooltip 和合理默认值表达语义，例如“HEX发送”默认不勾选表示普通文本发送。
- 后续沉淀：做 Qt 表格时把“首次可见宽度、列宽总和、用户拖拽保存、横向滚动兜底、复选框列提示”作为同一组设计检查。

## 2026-04-28：表格内置勾选框不会自动继承主界面 QCheckBox 样式

- 现象：多字符串表格里的发送、HEX发送、换行勾选框在白色表格背景上也是白色，未勾选时几乎看不出控件位置。
- 原因：`QTableWidgetItem` 的 `ItemIsUserCheckable` 由 item delegate 绘制，不等同于真正的 `QCheckBox` 控件；主窗口 QSS 里的 `QCheckBox::indicator` 高对比度样式不会稳定套到这些 item 勾选框上。
- 规避：需要和主界面勾选框保持一致时，在单元格中放置真实 `QCheckBox` 控件，并用对象名作用域复用同色系 indicator 样式；读写状态统一封装在 `rowCheckState()` / `setCheckItem()`，避免保存、轮询和立即发送分别读取不同来源。
- 后续沉淀：表格里的布尔列如果是核心操作，不要只依赖默认 item 勾选图标；要同时检查未勾选可见性、勾选颜色、hover 提示、键盘/鼠标操作和持久化状态。

## 2026-04-28：代码改动后必须同步文档和可复用经验

- 现象：功能修复完成后，如果只交付代码，后续代理容易重新踩同样的 Qt/CMake、布局、串口时序和配置持久化坑。
- 原因：代码只描述当前实现，不一定解释设计边界、排障路径和历史原因；后续代理读取上下文时会优先依赖 `AGENTS.md`、`README.md`、`docs/design.md` 和 `docs/pitfalls.md`。
- 规避：功能变化同步更新 README；架构或数据流变化同步更新 design；踩坑同步更新 pitfalls；非直观源码逻辑同步补中文注释；稳定经验整理到 `docs/skills/` 下的 skill 草案。
- 后续沉淀：已新增通用 Qt Widgets/C++ skill 草案 `docs/skills/qt-widgets-cpp/SKILL.md`，后续可迁移到全局 Codex skills 目录。

## 2026-04-28：Modbus Float32 不能按单个寄存器解析

- 现象：Modbus 读写工具里切换为 Float32 后，用户容易误以为每一行 16 位寄存器都能独立转换成浮点数；表格列宽如果只给固定值，首次打开还会出现右侧空白，看起来没有铺满窗口。
- 原因：Float32 是 4 字节数据，通常需要两个连续的 16 位寄存器合成；同时 Qt 表格在 `Interactive` 列宽模式下不会自动把所有列按窗口宽度铺满。
- 规避：Float32 解析必须从当前地址取当前寄存器和下一寄存器，缺少下一寄存器时显示占位；表格默认列宽按视口宽度初始化，并让最后一列填充剩余空间，刷新数据时不要覆盖用户拖拽后的列宽。
- 后续沉淀：Qt 表格里如果支持“单元格级数据格式”，要同时设计多寄存器格式的跨度、右键批量设置、用户拖拽列宽和 `QSettings` 持久化。

## 2026-04-28：Qt 程序不能只复制 exe 当便携版

- 现象：Qt Widgets 程序本机调试能运行，但把单个 `xray_serial_console.exe` 拷到别的目录或机器后可能打不开，常见报错是缺少 Qt DLL 或 platform plugin。
- 原因：Windows 下 Qt 应用依赖 Qt 动态库、`platforms/qwindows.dll`、样式/图片/翻译等插件，以及 MinGW 编译器运行库；这些不会自动跟随 exe。
- 规避：发布便携版时必须用对应 Qt 套件的 `windeployqt.exe` 扫描 exe 并拷贝依赖；安装包应基于已经部署好的便携目录生成，不要手工猜 DLL 列表。
- 后续沉淀：打包脚本应固定输出目录、版本号来源、Qt 路径参数、Inno Setup 可选路径，并把 `packages/` 等生成产物加入忽略列表。

## 2026-04-28：沙箱下 `windeployqt.exe` 可能无法启动 `qtpaths`

- 现象：运行打包脚本到 `windeployqt` 阶段时报 `Unable to query qtpaths: Error running binary qtpaths: pipe`。
- 原因：`windeployqt.exe` 内部还会启动 `qtpaths` 等 Qt 工具；受限沙箱可能允许直接运行 `qtpaths6.exe`，但阻止 `windeployqt` 再创建子进程。
- 规避：打包脚本要检查 `windeployqt` 退出码和 `platforms/qwindows.dll` 是否存在，失败时清理临时目录，不产出假便携包；真正发布打包在普通 PowerShell、Qt Creator 终端或本机开发终端执行。
- 后续沉淀：凡是 Qt 部署工具内部还会拉起子进程，Codex 沙箱验证只能证明脚本语法和错误处理，不能代替真实终端打包验收。

## 2026-04-28：复制到构建目录的脚本不能假设上级目录是源码根

- 现象：把 `scripts/package-windows.ps1` 复制到 Qt/CMake 构建目录后，如果仍用 `$PSScriptRoot` 的上级目录当源码根，会找不到 `include/appversion.h`、`README.md` 或写错输出目录。
- 原因：仓库内脚本路径是 `repo/scripts`，而构建目录路径通常是 `repo/build/<kit-name>`；两者相对源码根的位置不同。
- 规避：脚本需要优先支持 `-SourceDir`，并在构建目录运行时读取 `CMakeCache.txt` 里的 `CMAKE_HOME_DIRECTORY` 找回源码根目录。
- 后续沉淀：任何需要同时在源码目录和构建目录运行的工具脚本，都要显式区分“脚本所在目录、源码根目录、构建目录、输出目录”。

## 2026-04-28：BAT 只做入口，不要复制 PowerShell 里的打包逻辑

- 现象：为了方便双击执行，容易在 `.bat` 中重新写一套构建、部署、压缩和安装包逻辑。
- 原因：BAT 对路径、参数、错误码和中文输出处理都更脆弱；如果和 PowerShell 脚本各自维护逻辑，后续很容易行为不一致。
- 规避：BAT 只负责查找 `package-windows.ps1`、选择 `pwsh.exe` 或 `powershell.exe`、传递参数并暂停窗口；真实打包逻辑只保留在 PowerShell 脚本中。
- 后续沉淀：跨 shell 包装脚本应保持薄入口模式，复杂逻辑集中到更可靠、可解析验证的脚本里。

## 2026-04-28：工具箱页和独立工具不要混在菜单同一层

- 现象：计算器、文本 / HEX 转换、自定义协议组包解析都属于“协议与数据工具箱”的页签，而 Modbus 是独立读写窗口；如果工具箱页和 Modbus 都平铺在“工具”菜单下，用户看不出模块边界。反过来，如果 Modbus 当前只有一个读写工具却再套一层“Modbus”子菜单，也会显得层级多余。
- 原因：菜单结构没有表达对象关系，或把“模块分组”过度套用成单子项二级菜单。
- 规避：菜单按模块边界分组；工具箱页放入“协议与数据工具箱”子菜单；当前只有一个入口的独立 Modbus 读写工具直接放在“工具”菜单下。
- 后续沉淀：新增工具入口时先判断它是现有工具箱页、独立窗口、主界面开关还是发送动作；只有同一模块有多个并列入口时才增加模块子菜单。

## 2026-04-28：时间戳选项不能变成发送帧显示开关

- 现象：关闭“时间戳”后，主面板发送、轮询发送、多字符串发送或工具发送没有设备响应时，接收监视区看起来完全没有变化。
- 原因：`appendTrafficLog()` 在无时间戳路径里直接丢弃发送方向数据，把“是否显示时间”误当成了“是否显示发送帧”。
- 规避：时间戳只控制时间前缀；发送帧无论是否开启时间戳都要显示方向和内容。关闭时间戳时，发送帧仍用 `发→` / `TX->` 单独成行显示，接收方向继续保留原始流显示。
- 后续沉淀：设计串口日志选项时要拆开方向可见性、时间戳、HEX/文本格式、超时分包和暂停刷新，避免一个勾选项隐式改变另一个维度。

## 2026-04-28：PowerShell 脚本不会自动把外部命令失败变成脚本失败

- 现象：`cmake --build` 输出了 AutoMoc 错误，但调用 `scripts/build-local.ps1` 的外层工具仍看到 exit code 0。
- 原因：`$ErrorActionPreference = "Stop"` 只影响 PowerShell 错误，不会自动把原生命令的非 0 `$LASTEXITCODE` 转成脚本失败。
- 规避：封装外部命令调用，执行后检查 `$LASTEXITCODE`，非 0 时 `throw`；语法检查脚本也要在每个编译器调用后检查退出码。
- 后续沉淀：所有构建、检查、打包脚本都应显式传播原生命令退出码，否则验证结果不可信。

## 2026-04-29：构建目录缓存不能从旧项目直接复用

- 现象：Qt Creator 或 `cmake --build` 报 `No rule to make target 'xray_serial_console_autogen/mocs_compilation.cpp'`，或者 AutoMoc/AutoUic 仍到旧工程路径生成文件。
- 原因：构建目录里的 `CMakeCache.txt` 和 `CMakeFiles/` 仍记录旧源码根目录，例如 `CMAKE_HOME_DIRECTORY=D:/QT/projects/demo_widgets`；Makefile 会继续引用旧路径，当前仓库下的 `xray_serial_console_autogen/mocs_compilation.cpp` 就没有正确生成规则。
- 规避：同一个仓库可以复用构建目录，但不能复用旧项目的 CMake 缓存。遇到源码根目录不一致时，先备份或清理构建目录里的 `CMakeCache.txt` 和 `CMakeFiles/`，再用当前仓库重新运行 CMake 配置。
- 后续沉淀：排查 Qt/CMake 构建问题时，先检查 `CMakeCache.txt` 的 `CMAKE_HOME_DIRECTORY`、`CMAKE_PROJECT_NAME` 和 `CMAKE_PROJECT_VERSION`，确认构建目录确实属于当前仓库。

## 2026-04-29：PATH 里的 MinGW 可能覆盖 Qt 自带 MinGW

- 现象：Qt 安装目录实际存在，但构建脚本或终端仍找不到正确的 Qt Kit，或者使用了 `C:\mingw64\bin\g++.exe` / `mingw32-make.exe`。
- 原因：Windows PATH 中的通用 MinGW 可能先于 Qt Creator 管理的 `D:\QT\Tools\mingw1310_64\bin` 被发现；CMake 生成器、编译器和 Qt 库来自不同套工具链时，容易出现配置失败、链接失败或 AutoMoc 目标异常。
- 规避：本项目的 `scripts/build-local.ps1` 和 `CMakePresets.json` 固定传入 `CMAKE_PREFIX_PATH`、`CMAKE_CXX_COMPILER` 和 `CMAKE_MAKE_PROGRAM`。排查时优先查看 CMake 输出里的 CMake、CXX、Make 路径，确认都来自同一套 Qt 6.11 MinGW。
- 后续沉淀：如果后续需要支持多套 Qt 版本，应为每套 Qt 增加独立 preset 和构建目录，不要依赖全局 PATH 自动选择。

## 2026-04-28：开发构建目录不适合做唯一配置目录

- 现象：每次在 Qt Creator 构建后重新运行，波特率、超时分包勾选和发送区上次内容像是没有保存。
- 原因：程序曾优先把 `xray_serial_console.ini` 写到可写的程序目录；开发运行时程序目录通常是 CMake 构建输出目录，重新配置、清理或切换构建目录后，这个 INI 很容易丢失或换位置。同时发送编辑框文本原本没有纳入 `QSettings`。
- 规避：检测到程序目录位于 CMake 构建树内时，配置文件改用 `QStandardPaths::AppConfigLocation` 下的稳定目录，并从旧构建目录 INI 迁移一次；便携版目录不包含 CMake 构建标记时，仍可使用程序目录 INI。需要恢复的编辑框草稿要显式保存，例如 `send/editor_text`，不要只保存发送选项。
- 后续沉淀：持久化路径设计要区分开发构建、安装版和便携版；任何用户输入框是否需要恢复，都要作为配置字段逐项检查，不能假设“发送选项已保存”就代表发送内容也保存。
