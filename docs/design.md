# X-Ray Serial Console 设计文档

## 1. 文档目的

本文档用于说明 `xray_serial_console` 当前版本的整体设计思路、模块边界、线程模型、界面组织方式，以及后续演进方向，便于后续维护、功能扩展与提交评审。

## 2. 项目定位

X-Ray Serial Console 是一个基于 Qt Widgets 的本地串口调试台，主要面向以下场景：

- 串口联调
- 文本 / HEX 报文收发
- 协议验证
- 多命令轮询发送
- 文件透传与分块发送

当前实现以 Windows 为第一目标平台，串口层直接调用 Windows 原生 API，并使用 `SetupAPI` 与 `QueryDosDevice` 完成端口发现和端口描述补充。

## 3. 设计目标

### 3.1 功能目标

- 提供传统串口工具常见的高频功能，且操作路径足够短。
- 在一个主窗口内完成串口收发、文件发送、协议辅助和多字符串轮询。
- 提供中英文切换、设置持久化和状态反馈。

### 3.2 交互目标

- 保持较高信息密度，减少窗口切换成本。
- 重点操作要集中在主窗口下半区，便于持续联调。
- 发送链路要清晰区分主发送区、多字符串面板、协议工具箱三种来源。

### 3.3 工程目标

- 主界面逻辑和底层串口 I/O 解耦。
- 不依赖在线服务，保持本地可运行。
- 模块职责清晰，便于继续追加功能。

## 4. 总体架构

当前工程的代码已按用途整理到 `src/`、`include/`、`ui/`、`resources/` 和 `translations/` 目录中；本地参考截图统一放在 `docs/reference-images/`，避免根目录混杂。

当前工程主要由以下模块组成：

### 4.1 应用入口

- 文件：`src/main.cpp`
- 职责：
  - 启动 Qt 应用
  - 设置应用名、组织名、版本号
  - 应用全局调色板与基础样式

### 4.2 主窗口模块

- 文件：`include/mainwindow.h` / `src/mainwindow.cpp`
- 职责：
  - 构建主窗口界面
  - 管理菜单、状态栏、主发送区、主接收区
  - 连接主线程和串口工作线程
  - 管理显示设置、语言切换、文件发送流程
  - 调度多字符串面板、工具箱发送和 Modbus 实时监视

### 4.3 串口工作线程模块

- 文件：`include/serialportworker.h` / `src/serialportworker.cpp`
- 职责：
  - 在独立线程内打开、关闭和写入串口
  - 配置 Windows DCB 串口参数
  - 轮询串口接收数据与线路状态
  - 向主线程回传接收数据、错误和状态

### 4.4 应用配置模块

- 文件：`include/appsettings.h` / `src/appsettings.cpp`
- 职责：
  - 统一生成 `xray_serial_console.ini` 配置文件路径
  - 提供 `QSettings` 工厂函数
  - 首次生成 INI 时迁移旧的原生 `QSettings` 配置

该模块避免主窗口、多字符串面板等模块各自决定配置位置，确保窗口布局、串口参数、显示设置和多字符串预设落到同一个可追踪配置文件。

配置路径需要同时服务开发调试和便携发布：如果程序运行目录不是 CMake 构建目录且可写，优先使用程序目录下的 `xray_serial_console.ini`，便于便携版跟随目录迁移；如果检测到运行目录位于 CMake 构建树内，则改用 `QStandardPaths::AppConfigLocation` 下的稳定目录，并在首次创建时迁移旧构建目录 INI，避免 Qt Creator 重新构建或清理输出后丢失波特率、超时分包和发送草稿。

### 4.5 多字符串发送模块

- 文件：`include/multistringdock.h` / `src/multistringdock.cpp`
- 职责：
  - 维护多条可勾选发送项
  - 支持从主发送区捕获内容
  - 支持轮询发送与状态同步
  - 保存多字符串内容、启用状态、发送顺序、间隔和表格列宽

多字符串表格采用紧凑默认列宽，并在首次展开面板时由主窗口分隔条给右侧面板足够宽度，确保“发送、顺序、名称、内容、间隔、HEX发送、换行”等列默认可见。用户拖拽表头或分隔条后的状态继续通过 `QSettings` 保存；表头状态带版本号，避免旧版本把过宽默认列宽保存后一直恢复。发送、HEX发送、换行三列使用真实 `QCheckBox` 单元格控件，并复用主界面的蓝色勾选视觉，使未勾选边框和勾选状态都足够明显；状态读取仍统一经过 `rowCheckState()`，保证保存配置、立即发送和轮询发送逻辑不分叉。新增空白项默认按普通文本发送；从主发送区捕获内容时才沿用主发送区当前的 HEX / 换行设置。

### 4.6 协议与工具箱模块

- 文件：`include/toolboxdialog.h` / `src/toolboxdialog.cpp`
- 职责：
  - 提供协议拼包、解包和辅助计算功能
  - 输出可直接发送的独立报文

## 5. 线程与通信模型

### 5.1 线程划分

- 主线程：
  - 负责界面构建、用户交互、状态显示
- 串口工作线程：
  - 负责底层串口句柄、读写和轮询

### 5.2 通信方式

主线程不直接操作底层串口，而是通过 Qt 信号槽向 `SerialPortWorker` 发送请求：

- `requestOpenPort`
- `requestClosePort`
- `requestWriteData`
- `requestSetRtsEnabled`
- `requestSetDtrEnabled`

工作线程完成动作后再把结果回传给主窗口，例如：

- 串口打开成功
- 串口关闭
- 接收到新数据
- 写入字节统计
- 错误信息
- 调制解调器线路状态

### 5.3 设计收益

- 避免串口阻塞主界面
- 避免界面层直接持有复杂的系统串口状态
- 为后续扩展更高频场景留出空间

## 6. 界面设计

## 6.1 主界面分层

主窗口当前分为两大区域：

- 上部：接收监视区
- 下部：控制与发送区

其中下部控制区负责承载：

- 接收显示开关
- 多字符串状态摘要
- 超时分包配置
- 串口连接控制
- 串口参数配置
- 发送选项
- 校验附加选项
- 文件导入提示
- 主发送区与发送按钮

固定单位的数值输入框采用“标签 + 数字框 + 单位标签”的形式，例如接收超时、循环发送间隔、Modbus 扫描间隔、Modbus 日志超时和文件块间延时。单位不放进 `QSpinBox` 后缀，避免压缩数字编辑区域，也避免用户误以为 `ms` 是可编辑内容。

## 6.2 状态栏设计

底部状态栏采用分段式显示，当前主要展示：

- 品牌
- 发送计数
- 接收计数
- 串口摘要状态
- 线路状态

这样做的目的，是把联调时最常看的状态聚集到底部一行，减少频繁寻找信息的成本。

## 6.3 菜单设计

主菜单负责承载低频但必要的操作入口：

- 串口操作
- 视图操作
- 发送操作
- 工具入口
- 多字符串入口
- 语言切换
- 帮助与关于

多字符串入口已从低价值位置调整到顶栏，保证功能可发现性。
工具菜单按模块边界分组：“协议与数据工具箱”子菜单承载计算器、文本 / HEX 转换和自定义协议组包解析这些工具箱页；独立 Modbus 当前只有一个读写工具，因此直接作为“工具”菜单下的一级动作，避免出现只有单个子项的二级菜单。
帮助菜单保持两个面向用户的入口：“使用说明”用富文本集中说明基础收发、接收区筛选与超时分包、多字符串、Modbus、配置和打包入口；“关于”只展示当前应用版本、日期版本、发布日期和“首次发布”说明。版本文本统一从 `AppVersion` 读取，避免版本信息和帮助内容不一致。

## 7. 数据收发设计

## 7.1 接收链路

串口接收数据到达后，主窗口根据当前模式进行处理：

- 文本显示
- HEX 显示
- 时间戳附加
- 超时分包缓存
- 筛选关键字过滤
- 关键字高亮

超时分包模式下，会先缓存接收片段，等达到设定超时后再统一刷新，减少一帧数据被拆散显示的问题。主发送区、工具箱和多字符串轮询发送都会经过 `appendTrafficLog()` 记录发送方向；该函数在记录发送日志前会先刷新待显示的接收分包缓存，避免轮询发送不断重启接收超时计时器，导致主接收区长时间没有显示。Modbus 扫描请求可以作为主交通日志中的 TX 边界显示，但调用 `appendTrafficLog()` 时必须禁用主接收缓存刷新；工具自身的 TX/RX 原始日志继续使用独立的超时分包设置。这样主接收区和 Modbus 工具日志可以在同一串口数据流上按各自策略显示。时间戳只决定是否追加时间前缀，不作为发送帧显示开关；关闭时间戳时，发送帧仍以 `发→` / `TX->` 前缀单独成行显示，接收数据继续保持原始流显示方式。

接收区筛选关键字只影响主接收区显示，不影响收发计数和内部日志缓冲。筛选生效时，接收区标题会显示“筛选中”；如果当前缓冲内容被筛选后没有任何匹配行，接收区空白提示会说明正在按关键字过滤，并提示通过“视图 > 清除筛选”恢复全部显示。

## 7.2 主发送区链路

主发送区支持：

- 普通文本发送
- HEX 文本解析后发送
- 自动追加换行
- 附加校验字节
- 循环发送
- 发送区草稿保存与恢复

主发送区发送前，会根据当前设置统一构造最终发送字节流。

## 7.3 文件发送链路

文件发送设计分为三层：

1. 文件载入与预览
2. 发送参数确认
3. 分块发送执行

当前已支持：

- 发送预设方案
- 自定义每块字节数
- 自定义块间延时
- 发送进度条
- 中途取消
- 中断提示

该设计可以兼顾高速直发和单片机等慢速接收端。

## 7.4 多字符串与工具箱发送链路

多字符串面板和工具箱的发送都不强行复用主发送区的全部附加逻辑，而是走独立调用入口：

- `sendPresetPayload`
- `sendToolboxPayload`

这样做的原因是：

- 多字符串发送需要独立维护条目状态
- 工具箱中的协议报文通常已经完成组包和校验，不能再被主发送区重复叠加校验

Modbus 读写工具关闭或隐藏时，需要停止它自己的扫描定时器、取消正在等待响应的接收会话并清理监视缓存，避免隐藏窗口继续占用串口发送链路或接收主面板发送后的回包。Modbus 原始日志不是全局串口监听器，只有 `sendModbusMonitorRequest()` 发出读写请求并进入等待响应窗口后，`handleModbusMonitorBytes()` 才允许记录和解析接收字节；响应帧完成、超时、停止扫描、切换参数、清空数据或隐藏窗口都会结束这次会话。

Modbus 原始日志的时间戳、超时分包和超时值是独立设置，保存到 `modbus/log_*` 配置项。它们不能直接改写主接收区的 `display/*` 设置，否则关闭 Modbus 后会出现主接收区仍按 Modbus 日志策略缓存或过滤显示的错觉。反过来，Modbus 工具发送请求时追加的 TX 日志、窗口可见性保存、表格列宽保存和日志选项保存都不能触发主接收区的分包刷新；主接收区只由自身接收数据、明确的主发送链路边界、导出/关闭边界和自己的超时定时器决定何时落屏。`saveDisplaySettings()` 只负责持久化，不能隐式调用 `flushPendingReceivePacket()`。

## 7.5 Modbus 寄存器表解析设计

Modbus 读写工具的寄存器表默认按窗口宽度初始化列宽，同时继续使用 `Interactive` 表头模式，保证首次打开时地址、原始值、解析值、写入值、更新时间和状态列铺满表格，用户拖拽后的列宽仍通过表头状态保存。

Modbus 窗口的纵向 splitter 默认比例偏向原始收发日志，保证扫描时 TX/RX 帧、超时提示和解析异常不用一打开就滚动查找；寄存器表通过更协调的最小高度和固定行高保留必要可视行数，不用大片空白假装容量。顶部功能码、站号、地址、数量、扫描间隔等输入控件按实际内容设置宽度，功能码下拉框只保留常用显示余量，数字输入框宽度覆盖最大值、步进按钮和少量边距。标签和输入框作为字段组固定在一起，顶部参数行不使用尾部伸缩项消化剩余空间；默认窗口宽度和最小宽度按这些字段组反推，避免把“控件过宽”修成“右侧大留白”。

顶部“默认格式”下拉框只作为未单独设置地址的默认解析方式。用户在寄存器表中右键单元格时，可以把选中地址设置为独立格式，也可以一次选择多个单元格后批量设置。单地址覆盖按 Modbus 地址保存，而不是按当前表格行保存，这样起始地址或数量变化后，同一个地址再次出现时仍能使用原来的解析格式。

Float32 解析属于跨寄存器格式：`Float32 AB CD` 从当前地址和下一地址各取一个 16 位寄存器，按高字在前合成 4 字节浮点数；`Float32 CD AB` 使用相邻两个寄存器但交换高低字。若下一寄存器尚未返回有效数据，解析值显示为空占位，避免把半个浮点数误显示为有效值。

## 8. 配置持久化设计

当前通过统一 INI 配置文件保存用户常用设置，文件名为 `xray_serial_console.ini`。默认优先放在程序目录；如果目录不可写，则回退到系统应用配置目录。配置包括：

- 语言
- 显示模式
- 字体与字号
- 筛选与高亮关键字
- 超时分包开关与时间
- 波特率与串口参数
- 发送区草稿
- 文件发送分块与延时参数
- 多字符串列表、启用状态、发送格式与换行设置
- 主窗口位置与状态
- 多字符串分隔条与表格列宽
- Modbus 读写工具窗口位置、分隔条比例和寄存器表列宽；改变默认窗口几何时使用布局版本避免旧宽度配置覆盖新默认
- Modbus 默认解析格式与单地址解析格式覆盖
- Modbus 原始日志时间戳、超时分包和超时值

设计目标是让用户关闭程序后再次打开时，可以快速回到上次工作上下文。

## 9. Windows 打包设计

Windows 发布通过 `scripts/package-windows.ps1` 统一处理，脚本负责：

- 使用独立 Release 构建目录配置和构建 CMake 项目
- 从 `include/appversion.h` 或 `CMakeLists.txt` 读取版本号
- 调用当前 Qt 套件的 `windeployqt.exe` 收集 Qt 运行库、平台插件和编译器运行库
- 输出便携版目录与 ZIP 文件到 `packages/`
- 生成 Inno Setup `.iss` 脚本，并在本机存在 `ISCC.exe` 时继续编译安装包
- 复制到 CMake 构建目录后，通过 `CMakeCache.txt` 找回源码根目录，方便从 Qt Creator 的构建目录直接执行

`scripts/package-windows.bat` 只是可双击的包装入口，默认执行便携版打包；实际构建、部署和安装包生成逻辑仍集中在 PowerShell 脚本中，避免维护两套流程。

便携版是基础产物，安装包基于便携版目录二次生成。这样可以先保证“无需安装可运行”的发布方式稳定，再让安装包只承担快捷方式、安装路径和卸载信息等职责。

`packages/` 属于生成产物目录，不进入版本库。
打包流程的用户操作说明放在 `docs/packaging-windows.md`。

- 主要界面文案通过统一函数切换中英文文本

- 状态栏、菜单、关于窗口、多字符串面板和工具箱尽量同步切换
